# Condensed: Faiss Indexes

Summary: This tutorial covers implementing four Faiss vector similarity search indexes in Python: `IndexFlatL2`/`IndexFlatIP` (exact brute-force search), `IndexIVFFlat` (inverted file with k-means partitioning, requiring training and `nlist`/`nprobe` tuning), `IndexHNSWFlat` (graph-based ANN with `M`, `efConstruction`, `efSearch` parameters; no vector removal support), and `IndexLSH` (locality-sensitive hashing, poor for high dimensions). It helps with building nearest-neighbor search systems, choosing appropriate index types based on speed/accuracy/memory tradeoffs, configuring key parameters, and understanding training requirements and limitations of each index.

```python
import faiss
import numpy as np
np.random.seed(768)
data = np.random.random((1000, 128))
d = 128  # vector dimension
k = 3    # nearest neighbors to search
```

## 1. `IndexFlat*` – Exact Search

Stores vectors uncompressed with no training needed. Sequentially computes similarity against all vectors, guaranteeing globally optimal results. Two variants: `IndexFlatL2` (Euclidean) and `IndexFlatIP` (inner product).

```python
index = faiss.IndexFlatL2(d)
index.add(data)
D, I = index.search(data[:1], k)
```

**Best for:** Small datasets or when speed isn't critical. Perfect quality, poor speed.

## 2. `IndexIVF*` – Inverted File Index

Uses k-means to partition space into `nlist` cells. At query time, searches the `nprobe` closest cells, then finds `k` nearest neighbors within those cells.

| Parameter | Effect of Increasing |
|---|---|
| `nlist` | Faster search, but lower accuracy (edge/boundary misses) |
| `nprobe` | Better quality, slower speed |

**⚠️ Edge problem:** Query vectors near cell boundaries may miss nearest neighbors in adjacent cells if `nprobe` is too small.

```python
nlist = 5
quantizer = faiss.IndexFlatL2(d)
index = faiss.IndexIVFFlat(quantizer, d, nlist)
index.train(data)  # required before adding
index.add(data)

index.nprobe = 8  # set before searching
D, I = index.search(data[:1], k)
```

## 3. `IndexHNSW*` – Graph-Based Index

Builds a multi-layered proximity graph with "small-world" navigation properties.

| Parameter | Effect of Increasing |
|---|---|
| `M` | Better quality, linearly more memory |
| `efSearch` | Better search quality, slower search |
| `efConstruction` | Better graph quality, slower build time |

**⚠️ Does not support vector removal** (destroys graph structure). Memory-intensive due to graph storage. Best when RAM isn't a constraint.

```python
M = 32
index = faiss.IndexHNSWFlat(d, M)
index.hnsw.efConstruction = 32  # set BEFORE adding data
index.hnsw.efSearch = 16
index.add(data)
D, I = index.search(data[:1], k)
```

## 4. `IndexLSH` – Locality Sensitive Hashing

Hashes vectors into binary codes (maximizing collisions for similar vectors), compared via Hamming distance. `nbits` controls hash resolution.

**⚠️ Suffers curse of dimensionality** – even `d=128` is too large for good results. Not suitable for transformer-based embeddings. `nbits` must scale up with `d` to maintain quality.

```python
nbits = d * 8
index = faiss.IndexLSH(d, nbits)
index.train(data)  # required
index.add(data)
D, I = index.search(data[:1], k)
```

## Quick Reference

| Index | Training | Removal | Exact | Memory | Best Use Case |
|---|---|---|---|---|---|
| `IndexFlat*` | No | Yes | Yes | Low | Small data, baseline |
| `IndexIVF*` | Yes | Yes | No | Low | General-purpose ANN |
| `IndexHNSW*` | No | **No** | No | **High** | High quality when RAM available |
| `IndexLSH` | Yes | Yes | No | Low | Low-dimensional data only |
